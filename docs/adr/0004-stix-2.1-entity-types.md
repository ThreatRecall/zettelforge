# ADR-004: STIX 2.1 entity types

**Date:** 2026-07-02

**Status:** Accepted

## Context

ZettelForge's knowledge graph needed a typed ontology for CTI entities — threat actors, malware, vulnerabilities, indicators, and so on. We could have invented custom entity types tailored to our extraction pipeline, but the CTI ecosystem already standardizes on STIX 2.1, and analysts expect their tooling to speak it.

## Decision

Model knowledge graph entity types and relationship types on the STIX 2.1 standard (threat-actor, malware, tool, attack-pattern, vulnerability, campaign, indicator, infrastructure) rather than defining custom types, adding only a `zettel-note` bridge type for linking to the vector store.

## Consequences

**Pros:**

- Interoperability: entities map directly onto STIX objects, enabling export to and enrichment from platforms like OpenCTI.
- Analysts already know the vocabulary — no ZettelForge-specific ontology to learn.
- Relationship semantics (uses, targets, attributed-to, indicates, mitigates) come pre-defined by an industry standard instead of being invented ad hoc.
- Future integrations with TAXII feeds and other STIX-native tooling need no translation layer.

**Cons:**

- STIX 2.1 is verbose and broad; ZettelForge implements a pragmatic subset, which can surprise users expecting full spec coverage.
- Concepts that fall outside STIX (e.g., conversational notes) need bridge types, slightly diluting ontology purity.
- Tracking the standard means adapting if OASIS revises it.
